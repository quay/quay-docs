each repository requires a connection with either an _Organization_ or a _User_. This affiliation defines ownership and access control for the repositories.
 
== Organization account architecture

In {productname}, Organizations are architecturally implemented as a specialized type of user account. This design allows Organizations to share the same underlying account infrastructure as individual users, but it introduces specific considerations for email configuration:

* **Account email (internal)**: Organizations have an internal account identifier email that is part of the underlying user account table. This field enforces a global uniqueness constraint across all users and organizations in the {productname} deployment.

* **Contact email (recommended)**: Organizations have a separate `contact_email` field that is decoupled from the account identifier. This field is specifically designed for receiving automated system alerts, notifications, and recovery communications. Multiple organizations can share the same contact email address without conflicts.

[IMPORTANT]
====
When creating or managing organizations, use the `contact_email` field for notification purposes rather than relying on the account email. The contact email field:

* Does not enforce global uniqueness, allowing teams to use shared distribution lists.
* Prevents authentication conflicts when an email for an organization matches an email from a user.
* Ensures reliable delivery of quota warnings, security alerts, and other system notifications.
====

The distinction between these email fields addresses a common issue where organizations were forced to use dummy or fake email addresses to avoid account collision errors, which resulted in undelivered notifications.

== Tenancy model

image:178_Quay_architecture_0821_tenancy_model.png[Tenancy model]

* **Organizations** provide a way of sharing repositories under a common namespace that does not belong to a single user. Instead, these repositories belong to several users in a shared setting, such as a company.

* **Teams** provide a way for an Organization to delegate permissions. Permissions can be set at the global level (for example, across all repositories), or on specific repositories. They can also be set for specific sets, or groups, of users. 

* **Users** can log in to a registry through the web UI or a by using a client like Podman and using their respective login commands, for example, `$ podman login`. Each user automatically gets a user namespace, for example, `<quay-server.example.com>/<user>/<username>`, or `quay.io/<username>` if you are using {quayio}.

ifeval::["{context}" == "use-quay"]
* **Superusers** have enhanced access and privileges through the *Super User Admin Panel* in the user interface. Superuser API calls are also available, which are not visible or accessible to normal users. 
endif::[]

* **Robot accounts** provide automated access to repositories for non-human users like pipeline tools. Robot accounts are similar to {ocp} *Service Accounts*. Permissions can be granted to a robot account in a repository by adding that account like you would another user or team.
